# i18n GA — Internal Announcement

Plan: `dev-docs/plans/20260419-i18n-polish-and-announce.md` (Phase 4)
Release notes draft: `dev-docs/plans/20260419-i18n-release-notes.md`
Public blog post: `website/blog/2026-06-i18n-launch.md`

This is the internal, changelog-style record of what the i18n polish-and-announce
work shipped. It is the source artifact behind the public "VMark speaks your
language" post and the GitHub Release paragraph.

## TL;DR

VMark is now an international product, not just a technically-multilingual one:
the app auto-selects the right locale on first launch, and user-facing error
paths no longer leak English.

## What shipped

| Area | Change | Phase |
|---|---|---|
| First-run locale | New installs detect the OS language and pick the closest supported locale instead of always defaulting to English | 1 |
| Upgrade safety | Existing users keep their persisted `general.language` — auto-detection only applies when there is no saved value | 1 |
| Live switching | **Settings → General → Language** still changes the locale live, no restart | (existing) |
| Rust error i18n | User-facing backend error messages route through `rust-i18n` and translate to the active locale | 2 |
| Error namespace | Added an `errors:` top-level namespace to all 10 `src-tauri/locales/*.yml` files | 2 |
| Translation workflow | The `translate-docs` skill gained an App-string mode that syncs new keys across all 10 locales with placeholder preservation and cultural polish | 3 |
| Announce | Public blog post, blog index entry, and this GA artifact | 4 |

## Locales shipped

All builds are a single multilingual binary — every locale is always present.

| Language | Code |
|---|---|
| English | `en` |
| Simplified Chinese (简体中文) | `zh-CN` |
| Traditional Chinese (繁體中文) | `zh-TW` |
| Japanese (日本語) | `ja` |
| Korean (한국어) | `ko` |
| German (Deutsch) | `de` |
| Spanish (Español) | `es` |
| French (Français) | `fr` |
| Italian (Italiano) | `it` |
| Brazilian Portuguese (Português do Brasil) | `pt-BR` |

## OS-detection fallback chain

The first-launch resolver matches `navigator.language` / `navigator.languages`
against the 10 supported codes:

| OS preference | Resolves to |
|---|---|
| `zh-CN`, `zh-SG`, `zh-Hans-*` | `zh-CN` |
| `zh-TW`, `zh-HK`, `zh-MO`, `zh-Hant-*` | `zh-TW` |
| `pt-BR`, `pt-PT`, `pt` | `pt-BR` |
| Exact `en`, `ja`, `ko`, `de`, `es`, `fr`, `it` (and base-language match) | that locale |
| Anything else (e.g. `ru`, `ar`) | `en` |

## What deliberately stays English

- Developer diagnostics — `console.*`, debug loggers, panic messages, log-file
  entries.
- Internal plumbing errors (mutex poisoning, channel-closed, "BUG: …").
- Technical tokens inside translated strings — keyboard shortcuts (`Cmd+S`),
  product names (`Markdown`, `Pandoc`, `YAML`), and `%{…}` / `{{…}}` placeholders.

## How to contribute translations

1. **Source of truth is English.** Edit only the English files:
   - React UI strings: `src/locales/en/*.json`
   - Rust backend strings: `src-tauri/locales/en.yml`
   - Website pages: `website/**/*.md` (English originals)
2. **Propagate with the skill.** Run the `translate-docs` skill (App-string mode
   for the locale files, default mode for website pages). It translates only the
   missing keys, preserves placeholders verbatim, applies CJK fullwidth
   punctuation, and never reorders or renames existing keys.
3. **Native-speaker polish welcome.** If a machine-assisted translation reads
   awkwardly, edit the target locale file directly — keep the key identical to
   English and preserve every placeholder.
4. **Report leaks.** English surfacing in a non-English session on any
   user-facing path is a bug: open an issue at
   <https://github.com/xiaolai/vmark/issues>.
