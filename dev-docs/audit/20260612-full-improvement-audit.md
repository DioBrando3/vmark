### H5. MCP checkpoint 'append' rewrites the whole history file per AI write and races with itself, losing restore points

**Dimension:** State management | **Kind:** bug | **Effort:** small
**Files:** `src/stores/mcpCheckpointPersistence.ts:100-118`, `src/hooks/mcpBridge/v2/document.ts:146-156`, `src/hooks/mcpBridge/v2/selection.ts:232-242`, `src/hooks/mcpBridge/v2/workflow.ts:247-257`

**Evidence:** appendCheckpoint() at mcpCheckpointPersistence.ts:100 does `existing = await readTextFile(path)` then `await writeTextFile(path, existing + JSON.stringify(cp))` — a full read-modify-write of the entire JSONL (budgeted up to 5 MB of contentBefore snapshots) on EVERY MCP write. The file header claims 'Append-only JSONL keeps push cheap (one fs write per checkpoint)' — the code does not append. All three callers fire it as `void appendCheckpoint(cp)` (unawaited), so two back-to-back MCP writes interleave: B reads before A writes, B's write drops A's line. The in-memory store still has the checkpoint until restart; after restart the restore point is silently gone. This history is explicitly the user's safety net for AI edits (McpHistoryButton.tsx header: 'Replaces the safety net the deleted suggestions feature used to provide').

**Recommendation:** Use tauri-plugin-fs writeTextFile with { append: true } for the hot path (true O(1) append), and serialize all writers (append/rewriteAll) through a single promise queue like the inFlightSync chain already used in settingsStore/shortcuts.ts:397-401. Add a test that fires two checkpointPush+append cycles concurrently and asserts both lines land on disk.

**Verifier (high confidence):** Every factual claim verified against source. (1) Evidence accurate: appendCheckpoint (src/stores/mcpCheckpointPersistence.ts:100-117) is a full read-modify-write (readTextFile of entire file, concat, writeTextFile of entire file) despite the header (lines 2-5) claiming cheap append-only JSONL; all three cited callers fire it unawaited via `void appendCheckpoint(cp)` (document.ts:156, selection.ts:242, workflow.ts:257); CHECKPOINT_TOTAL_BYTE_LIMIT is 5 MiB (mcpStore.ts:44); McpHistoryButton.tsx:6-7 contains the quoted safety-net line verbatim. The lost-update race is real: B's readTextFile before A's writeTextFile drops A's line, and after restart hydrate reads only disk, so the restore point silently vanishes. (2) No mitigation exists: the bridge dispatcher (mcpBridge/index.ts:115) fires handleRequest unawaited so requests are not serialized; rewriteAll is also void-fired from hydrate and McpHistoryButton.tsx:135, adding more unserialized writers; the persistence tests (mcpCheckpointPersistence.test.ts) mock fs entirely and only test single sequential calls — the 'appends a new line' test actually pins the read-modify-write, not true append; no platform cfg or upstream guard applies. Race window is realistic: the append does 3 Tauri IPC round-trips moving up to 5+ MB while the MCP client's next localhost tool call arrives much faster; disk file can also exceed the 5 MB budget between restarts since compaction only happens on hydrate. (3) Recommendation is sound and convention-compliant: plugin-fs ^2.4.5 (package.json:66) supports writeTextFile { append: true } (stable v2 API; node_modules read was permission-blocked but the API predates 2.4.5); the referenced inFlightSync promise-chain exists exactly at settingsStore/shortcuts.ts:397-401; the suggested concurrency test matches the TDD rules. ## Dimension health summaries

Each auditor's overall assessment, including sub-threshold one-liners that did not become findings.

### Architecture

Architecture health: the skeleton is genuinely good — zero circular dependencies across 2,130 modules (verified by running depcruise), AppShell.tsx and EditorArea.tsx are confirmed pure composition roots per ADR-007 (only react-i18next imports), no ../../../ chains exist in production code, utils/markdownPipeline and utils/cursorSync are leaf-pure, services/ is cleanly organized into 12 domain folders, and src/shared/ is a coherent single-purpose Rust↔TS contract directory. The problem is that the written constitution (ADR-013, architecture.md) and the enforcement machinery (.dependency-cruiser.cjs) have both decoupled from reality: 15 utils files import Tauri/services, 12 services files import hooks (inverted layering), 10 services files use React, and the cruiser passes all of it green because its rules only encode a fraction of the documented tier model while architecture.md claims full enforcement. The single sharpest bug found: the menu-ID contract generator still reads the deleted src-tauri/src/menu.rs, so src/shared/menu-ids.json has silently drifted 30+ IDs behind the real menus since February, neutering actionRegistry's drift validation. Items that did not make the cut: hooks count drift in architecture.md (doc says 100, actual 94 non-test files) and src-tauri file count (24 vs 33) — both folded into the doc-drift finding; plugins/codemirror/index.ts is a 29-export barrel but is explicitly documented as a module cluster, acceptable; feature-local stores in components/ (quickOpenStore, commandPaletteStore) follow the 'keep features local' rule and the stores-no-import-components cruiser rule correctly only scopes ^src/stores/; src/utils/icons.tsx is a React-flavored file in utils but has no react import (JSX automatic runtime), borderline; lib/ (pty.ts, ghaWorkflow, markdownLinkCheck) imports Tauri at 9 sites but lib/ has no documented tier position at all — fix by assigning lib a tier when amending ADR-013; the workspace facade's tolerate-missing-provider fallback in useWorkspace.ts is sensible test ergonomics, not a hole.

### State management

Zustand health: STRONG overall — this is one of the most disciplined store layers I have audited. Verified clean: zero component destructuring violations (`const {x} = useXStore()` grep returned nothing across ~960 non-test files); selectors are consistently primitive-returning field picks (FindBar, ContentSearch, Tab, App), so the total absence of useShallow is correct, not a gap; getState() is consistently confined to callbacks/handlers; every keyed update I read is guarded per rule 50 (documentStore.updateDoc:145-158 plus the setTabExistenceGuard defense-in-depth, lint/unifiedHistory/revision/largeFileSession all no-op on missing keys); async races are systematically token-guarded (contentSearchRequestId, lint linkCheckTokens, fileLoad loadId, aiStore _detectId/_loadId); settingsStore is exemplary persist engineering (versioned migrate, recursive zero-trust sanitization, clamped numerics) and aiProviderStore moves API keys to the keychain with status-aware hydration; the plain-.subscribe-with-manual-prev rule is honored everywhere I sampled (i18n.ts:91, lint.ts:194, focusMode, search, sourceMermaidPreview, useTerminalSessions); per-tab cleanup is centralized in hooks/tabCleanup.ts and called from all close/detach paths. The real issues are concentrated in the seams: the MCP checkpoint 'append' that actually rewrites the whole file unserialized (top finding), six persisted stores missing the validation standard the team itself defined, and the T09 consolidation stopping halfway (oversized god-store files + 62 consumers still on legacy shims). Did not make the cut: tabStore.closeTab fires toast.info inside the set() updater (tabStore.ts:279 — impure updater, works because Zustand calls it once); uiStore searchReplaceCurrent/All dispatch window CustomEvents from store actions (uiStore.ts:478-483 — store→DOM coupling); mcpStore byteSize uses contentBefore.length (UTF-16 code units, not bytes — CJK-heavy histories under-count the 5MB budget ~3x in UTF-8 terms, mcpStore.ts:224); tabStore.untitledCounter never resets so Untitled-N grows monotonically per session; lint.ts resolves a Promise inside a set() updater (lint.ts:120-133 — works, but couples set to external control flow); workflowStore keeps a Set in state (correctly copy-on-write at :318-324); settingsTypes.ts at 451 lines is type-only and lowest priority of the oversized files; commandPaletteStore lives under components/ rather than src/stores/ (defensible feature-locality, but inconsistent with the other 30+ stores).

### Editor core

EDITOR CORE health: architecturally strong — plugin cleanup discipline is genuinely good (all six WYSIWYG popup views, MathInlineNodeView, aiSuggestion's view(), columnResize and the multiCursor module all tear down listeners/observers/timers correctly; TiptapEditor's unmount effect cancels every timer and flushes pending content first per #755), transaction perf is mostly well-engineered (codePreview's three-tier fast paths, CJK letter-spacing's step-map incremental updates, footnote's no-footnote cache, debounced search), and plugin structure follows rule 50 with CSS properly co-located (editor.css even points footnote styles back to the plugin dir). The standout risk area is the markdown serializer's string post-processing block (serializer.ts:264-283): three independent defects — a range-blind &#x20; global replace, over-aggressive escape stripping, and a fence regex that misses list-indented code — each verified mechanically with the project's own remark modules, and all amount to silent content mutation on the parse→serialize loop that runs on every WYSIWYG keystroke. ### Rust backend

RUST/TAURI BACKEND: surprisingly healthy for its size — this is one of the most disciplined Tauri backends I have audited. Panic discipline is near-perfect: of 522 unwrap/expect/panic hits, essentially all live in #[cfg(test)] modules; the handful in production code are static-regex/invariant expects (workflow/template.rs:28,66,69; the documented-infallible updater header at lib.rs:601). ### Security

VMark's Rust attack surface is, on balance, carefully built: the debug automation bridge (9323) and debug_log command are correctly cfg(debug_assertions)-gated; the production MCP bridge binds 127.0.0.1 only, enforces a token auth phase before any request, bounds queues, and serializes writes; secrets live in the OS keychain (secure_store.rs); external_editor.rs has a thorough XSS-supplied-override gate; CLI AI providers pass prompts via argv (no shell interpolation); pty_spawn requires an absolute, existing shell; and connect-src defaults to 'self', which blocks fetch-based exfiltration. The weak spots cluster in three areas: (1) the AI/genie/workflow pipeline feeds untrusted document content into autonomous CLI agents (codex exec / claude -p) with no trust boundary; (2) capability/CSP scoping is broader than least-privilege needs (img-src http:/https: any host, assetProtocol **/*, fs $HOME/**); (3) a couple of input-validation gaps (workflow save-file symlink escape, document link schemes not allowlisted). Lower-priority items not broken out below: the MCP bridge never validates the WebSocket Origin header (mitigated by the random OS-assigned port + 64-hex token a browser cannot read, server.rs:200); withGlobalTauri:true (tauri.conf.json:13) exposes the full IPC surface on window.__TAURI__ so any future XSS reaches every registered command (partially contained by script-src 'self'); strip_workspace_quarantine_cmd (quarantine.rs:134) trusts a frontend-supplied root and removes Gatekeeper quarantine on depth-1 supported files without verifying the path was user-selected; and style-src 'unsafe-inline' (tauri.conf.json:29) permits CSS that can pull remote background images over the same img-src channel.

### CSS / design tokens

CSS/token health is genuinely strong: check-design-tokens.mjs reports ZERO hex warnings today, z-index is fully tokenized (raw values exist only in the intentionally standalone export reader bundle), the rgba-fallback-before-color-mix pattern is applied consistently (lint.css, StatusBar.css, GenieResponseView.css, title-bar.css), no [data-theme] selectors remain in app CSS, popup duplicates across files are properly namespace-scoped, and the ADR-014 theme unification (typed catalog -> legacy var projection with frozen regression snapshots) is well executed. The real problems cluster in two places: (1) the dark-theme delivery layer — several legacy vars the CSS consumes (--hover-bg, --popup-shadow, --contrast-text) are deliberately never flipped in dark mode, pushing per-file .dark-theme overrides that roughly half the consumers forgot; and (2) a removed --spacing-* token family that still has live consumers and stale documentation. One-liners that did not make the cut: workflow-editor.css is 1099 lines and StatusBar.css 819 vs the ~300-line split guidance; title-bar-input focus indicator is a gray alpha change 0.4->0.6 (color-only, low perceivability vs rule 33); .dark-theme .drop-zone-indicator (drop-zone.css:50) hardcodes rgba(90,168,255,0.12) instead of var(--accent-bg); markmap's .mermaid-export-btn:focus-visible indicates focus via opacity only (markmap.css:96-99); --accent-text emitted by computeCoreColorVars (useTheme.ts:178) has zero CSS consumers (dead var); rule 32's scrollbar section still documents 1px scrollbars while index.css:289-292 ships 10px overlay scrollbars (#946); App.tsx:68-71 error-boundary hardcodes #dc2626/#fef2f2 (defensible since the theme system itself may have crashed, but it is invisible to every lint gate).

### Internationalization

I18n health: structurally strong, operationally leaky. The architecture is genuinely good — i18next v25 with lazy per-namespace loading and zh-TW→zh-CN→en fallback (src/i18n.ts), rust-i18n with a single localized menu builder, a key-parity + JSON-placeholder gate wired into check:all, and dedicated propagation scripts (propagate-i18n-keys.mjs, propagate-rust-errors.mjs, find-untranslated.mjs). React component code is remarkably clean: a repo-wide grep found essentially zero hardcoded English JSX attribute strings in src/components and src/pages. The leaks are all in places the gate can't see: (1) a real plural-resolution bug (`_plural` legacy suffix dead under i18next v25) that makes close-window/save dialogs grammatically wrong in every locale; (2) the Universal Toolbar rendering hardcoded English while 66 fully-translated toolbar.* keys sit dead in 10 locale files; (3) the entire workflowEditor namespace shipped as verbatim English copies in all 9 locales, with its diagnostics keys both dead and using broken single-brace placeholders; (4) table/tab context menus with no keys at all; (5) imperative-DOM plugin UI (previews, source peek, TOC, image errors) bypassing t(); and (6) 187 raw-English Rust error strings vs 36 t!() calls, some toasted directly to users. The lint itself has blind spots: no YAML %{var} placeholder validation, stale keys only warn, untranslated-value detection exists (find-untranslated.mjs) but is not in the gate. Did not make the cut: src/locales/en/menu.json is an empty {} stub (dead namespace file — delete or document); escapeValue:false is safe since no t() output reaches dangerouslySetInnerHTML (only typed icon SVGs and sanitized renders); WorkflowEditor technical placeholders (placeholder=\"job-id\", \"github.event_name == 'push'\") are acceptable as code examples; CJK locales carrying useless-but-harmless `_one` plural keys is a side effect of strict key parity; statusbar word-count popover and shortcuts settings both use proper translation-with-fallback patterns worth copying (settingsShortcutLabels.ts); fr/export.json's 3 stale keys are the only current parity drift.

### MCP stack

MCP stack health is good overall: the post-pruning architecture is clean and well-documented. Every webview handler in the v2 dispatcher (session 1, workspace 7, document 3, workflow 2, selection 2 = all 15 routed types in v2/dispatch.ts) goes through wrapHandler, which guarantees respond() on both success and thrown-error paths — ZERO violations of the .claude/rules/50 try/catch+respond contract; malformed JSON args and shape-invalid payloads are also answered or dropped loudly at the entry point (index.ts:84-107). Version constants are in sync (0.8.4 in vmark-mcp-server/package.json, cli.ts VERSION, and root package.json). website/guide/mcp-tools.md accurately documents the current 5-tool/15-action surface including the error-envelope table — no dead or undocumented tools in the shipped sidecar. Optimistic concurrency (expected_revision/STALE), checkpointing, pendingSave coordination, the bounded per-client outbound queue with force-disconnect on response loss, and the auth handshake with timeout are all solid engineering; test coverage is extensive on all three layers. The weak spot is timeout/retry semantics: the Rust wake-and-retry re-emits writes with no webview-side request-id dedup (double-apply of selection.set/apply_patch when expected_revision is omitted), and the sidecar gives up at 10s while Rust retries until 20s, making the recovery path invisible to AI clients. Cut one-liners: cli.ts:470 reports MCP serverInfo version '0.1.0' while --version says 0.8.4 (intentional per server.ts:59 comment but confusing in client UIs); state.rs:181-219 and server.rs:831-880 still carry the legacy v1 operation names (document.getContent, windows.list, etc.) that the pruned sidecar never sends — dead allowlist/fast-path code; respond() in src/hooks/mcpBridge/utils.ts:26-29 swallows invoke failures, silently feeding the Rust timeout/retry path; workspace.open (v2/workspace.ts:74-107) doesn't dedupe an already-open filePath, creating duplicate tabs for the same file; payload size is bounded only by tokio-tungstenite's 64 MiB default with no app-level cap on document.write; the mcp_bridge_heartbeat exists but is_webview_alive only feeds a log line.
