## S0.2 — Programmatic embed (the headline unknown)

Verified `@slidev/cli` exports `createServer` and `resolveOptions` as functions.
A real script (`probe.mjs`) ran:

```js
const options = await resolveOptions({ entry: "slides.md" }, "dev");
const server  = await createServer(options, { server: { port: 0, host: "127.0.0.1" }, logLevel: "silent" });
await server.listen();
// fetch http://127.0.0.1:<port>/ → 200, html contains id="app"
await server.close();
```

Result: **BOOT_MS: 341, STATUS: 200, HAS_APP_MOUNT: true, CLOSED_OK**. Points at
an arbitrary `slides.md` (no scaffolded project needed). **The "no prior art for
embedding Slidev" risk is retired.**

Caveat recorded: Slidev's `getRoots()` reads `process.argv[1]`; it must run in a
real spawned process (the content-server CLI), not an `-e` eval. The supervisor
spawns a script, so this is satisfied. Middleware-mode/HMR-proxy under the KB
origin (ADR-9) still to be wired in Phase 6; the standalone sub-port + same
cookie fallback is the safe default.

## S0.3 — Export

`build`/`exportSlides` are **not** programmatic exports — export is the
`slidev export` CLI subcommand, which depends on `playwright-chromium`
(optional peer, ~150 MB, per-OS). Decision: the supervisor spawns
`slidev export` as a child process (not a programmatic call) and provisions
Chromium on first export (Phase 7). Actually downloading Chromium + running a
headless export here was deliberately skipped — it is heavy and the remaining
risk (Chromium provisioning + macOS quarantine of the downloaded browser) is the
same external-infra question as S0.5, not a Slidev-API question.

## S0.5 — Bundle size + provisioning

Measured: a fresh `npm install @slidev/cli@latest` = **668 packages, 451 MB**
node_modules (Node 22; Slidev `engines.node: >=20.12.0` — confirms D3.1).

- This is the **Slidev bundle** (ADR-2, download-on-first-use). The base KB
  bundle is far smaller (Hono + unified + jsdom).
- `@yao-pkg/pkg` is unusable for this (confirmed Vite/Vue tree) → ADR-2's
  CI-built signed tarball stands.
- **External-infra residue (cannot PASS in a dev shell):** codesigning/notarizing
  the in-bundle Node binary, hosting the signed tarball, and macOS Gatekeeper
  quarantine handling of the extracted tree + the downloaded Chromium. These are
  Phase 1 + release-pipeline work requiring Apple Developer signing infra.

## Net

The two Slidev *software* unknowns (programmatic embed; export mechanism) are
resolved. What remains is **packaging/trust infra**, which is external to code
and tracked in ADR-2 + Phase 1.
